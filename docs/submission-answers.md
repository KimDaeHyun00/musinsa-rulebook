────────────────────────── 문항 1 (현재 500자 / 800) ──────────────────────────

musinsa-rulebook은 한 개발자가 익힌 코딩 교훈을 캡처하여, 팀 전체의 Codex 에이전트가 자동으로 따르고 모든 변경을 점수로 검수하게 만드는 '팀 공용 룰북' Codex 플러그인입니다.

대상 사용자는 이미 Codex로 개발하는 무신사 백엔드 개발자입니다. 무신사는 전사가 AI 에이전트로 코딩하지만 각자 분리된 세션에서 작업하기 때문에, 한 사람이 깨달은 교훈이 팀 전체로 전파되지 않습니다. 그 결과 600줄 쿠폰 쿼리나 인덱스 무력화처럼 무신사가 기술블로그에 이미 공개한 교훈조차 문서로만 남아, 에이전트는 이를 알지 못한 채 같은 실수를 반복합니다. 개발자가 매번 팀 규칙을 직접 일러주어야 하는 지점에서 생산성이 막힙니다.

이 플러그인은 버그를 고친 직후 /remember로 교훈을 한 번 등록하면(별도 입력 없이도 직전 작업에서 규칙을 추론합니다), 이후 PR 직전에 /review가 모든 팀원의 변경을 그 룰북으로 점수와 함께 검수하도록 하여 이 단절을 메웁니다.

────────────────────────── 문항 2 (현재 531자 / 800) ──────────────────────────

이 문제를 선택한 이유는, AI 네이티브로 전환한 조직일수록 개발 지식이 개인 세션에 갇혀 팀으로 축적되지 않는 단절이 커지기 때문입니다. 과거 코드리뷰와 멘토링으로 전파되던 시니어의 판단이, 각 개발자가 AI와 일대일로 작업하면서 파편화됩니다. 무신사의 공개 자료가 이를 두 측면에서 입증합니다.

첫째, 무신사는 AGENTS.md 파일로 규칙을 세션마다 읽어 생성 코드에 적용한다고 공개했으나, 이 파일은 수동으로 관리되어 한 개발자의 교훈이 자동으로 반영되지 않습니다. 둘째, ORDER BY의 DATE_FORMAT으로 인덱스가 무력화되어 p99 응답을 350ms에서 20ms로 개선한 사례, 600줄 god-query, 타입 분기, 단일 Redis 장애 등 값비싸게 얻은 교훈이 모두 블로그 글로만 존재합니다.

즉 무신사 개발자는 교훈은 있으나 에이전트가 그것을 강제할 형태가 없는 지점에서 막힙니다. 무신사가 채용 면접에서 Codex 코드를 자체 AI 평가 도구로 채점하듯, 이 플러그인은 그 채점을 현직 개발자의 PR 시점으로 옮깁니다.

────────────────────────── 문항 3 (현재 860자 / 1000) ──────────────────────────

플러그인은 계층형 룰북을 기반으로 동작합니다. 규칙은 seed(플러그인에 동봉된 무신사 공개 교훈), org(선택적 조직 공통 규칙), repo(.musinsa/rules.yaml, 팀이 직접 키우는 규칙) 순서로 합쳐 읽으며, 하위 계층이 상위를 덮어씁니다. 플러그인 코드와 규칙 데이터가 분리되어 있어, 규칙은 재배포 없이 레포 파일로 축적되고 공유는 평소의 git/PR 흐름을 그대로 따릅니다.

두 개의 스킬로 구성됩니다. /remember는 교훈을 구조화된 규칙(탐지 정규식, 판단 힌트, 근거, 수정안, 출처)으로 변환해 룰북에 추가합니다. 규칙을 직접 입력하면 그대로 반영하고, 인자 없이 호출하면 최근 git diff와 대화에서 무엇을 어떻게 고쳤는지 분석하여 규칙을 추론합니다. /review는 변경분을 룰북과 대조해 검수합니다. 정규식으로 탐지 가능한 위반은 check.py가 결정론적으로 판정하여 동일 코드에 항상 동일 점수를 보장하고, 설계상의 문제는 에이전트가 규칙의 판단 힌트로 평가합니다. 100점에서 위반 심각도에 따라 차감하며 block이 하나라도 있으면 게이트를 BLOCKED로 막습니다. 판단 기준은 룰북 그 자체이며, 모든 seed 규칙은 무신사 블로그 출처와 수치를 함께 제시합니다.

정보가 부족하거나 판단이 어려운 경우의 동작도 정의했습니다. /remember가 규칙을 명확히 추론하지 못하면 임의로 작성하지 않고 한 가지만 되묻습니다. 추론한 규칙은 기록 전 사용자 확인을 거쳐 오추론을 방지합니다. check.py 의존성이 없거나 실패하면 조용히 건너뛰지 않고 그 사실을 명시한 뒤 수동 판단으로 대체합니다. 근거가 애매한 위반은 block이 아니라 info로 분류해 과도한 차단을 피합니다.

────────────────────────── 문항 4 (현재 598자 / 800) ──────────────────────────

AI에 맡긴 작업과 직접 판단한 부분을 구분했습니다. AI에는 무신사 기술블로그(61개 글 목록화, 핵심 20여 편 정독), Codex 공개 사례, 경쟁 도구 조사를 다중 에이전트로 수집·구조화하는 일과, 각 규칙의 정규식 및 수정안 초안, 문서 작성을 맡겼습니다. 무엇을 규칙으로 삼을지, 점수와 게이트 설계, 결정론 채점과 LLM 판단을 결합한 하이브리드 구조, 인자 없는 추론 모드의 도입 여부는 직접 판단했습니다.

막혔던 지점도 있었습니다. 무신사 블로그가 Cloudflare로 차단되어 본문을 읽지 못했고, Wayback Machine 스냅샷으로 우회해 전문을 수집했습니다. git 배포 설치 시 경로가 올바른지 확신할 수 없어, 실제 Codex CLI에서 마켓플레이스 등록부터 설치까지 직접 검증했습니다.

받아들이지 않은 AI 제안도 있습니다. 점수를 LLM이 직접 매기게 하자는 방향은, 같은 코드에 점수가 흔들리면 현업이 신뢰하지 않으므로 거부하고 정규식 결정론 채점을 핵심에 두었습니다. 또한 리뷰 출력에서 에이전트가 근거를 임의로 요약해 17배 같은 무신사 수치를 누락하기에, 요약을 금지하고 check.py 결과를 그대로 출력하도록 강제했습니다.

────────────────────────── 문항 5 (현재 638자 / 800) ──────────────────────────

입력에서 결과로 이어지는 예시는 다음과 같습니다. ORDER BY DATE_FORMAT, SELECT *, 4개의 JOIN을 가진 쿠폰 Mapper와 타입 분기 배송 서비스를 입력하면, 실제 Codex에서 Musinsa Readiness 53/100과 게이트 BLOCKED, 그리고 위반 5건이 파일·라인, 근거(p99 17배), 수정법, 무신사 출처와 함께 출력됩니다.

정상과 예외를 모두 확인했습니다. 정상 경로에서는 위 BLOCKED가 나오고, 팀 규칙을 추가하면 85점으로 등록자까지 표시됩니다. 예외로 위반이 없는 BigDecimal 코드는 100점 PASS로 처리되며, 스킬 이름을 부르지 않고 검수해 달라는 자연어만으로도 스킬이 자동 선택됩니다.

의심한 부분은 LLM 점수의 변동성과 출처 URL 훼손이었습니다. 전자는 결정론 check.py로 고정했고, 후자는 일부 글자가 어긋나도 끝의 ID로 링크가 열리는 것을 확인했습니다. 테스트 중 타입 분기 정규식이 getType() 괄호에서 빗나가 패턴을 수정했고, 수치 누락을 막기 위해 출력을 3단 구조로 재설계했습니다. 아직 부족한 점은 seed 규칙이 6개로 적고, 조직 공유 계층이 미구현이며, 추론 모드의 심각도 판단이 실행마다 달라 확인 단계가 필요하다는 것입니다.
