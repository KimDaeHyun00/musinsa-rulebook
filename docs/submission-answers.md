## 문항 1 — 무엇을, 누가, 어떤 상황에서 쓰나요? (≤800자)

**한 줄 요약:** musinsa-rulebook은 한 개발자가 배운 코딩 교훈을 캡처해, 팀 전체의 Codex 에이전트가 자동으로 따르고 PR을 점수로 채점하게 하는 '팀 공용 룰북' Codex 플러그인입니다.

**누가** — 이미 Codex로 개발하는 무신사 백엔드 개발자.
**어떤 상황** — 무신사는 전사가 AI 에이전트로 코딩하지만 각자 사적인 세션에서 일해, 한 사람이 깨달은 교훈이 팀으로 전파되지 않습니다. 그래서 ⓐ 누군가 버그를 고친 직후, ⓑ PR을 올리기 직전 두 순간에 씁니다.
**어디서 막혀** — 600줄 쿠폰 쿼리, 인덱스 무력화처럼 무신사가 블로그에 공개한 교훈이 '글'로만 존재해, AI 에이전트는 이를 모른 채 같은 실수를 반복합니다. 사람이 매번 "우리 규칙 지켜"라고 일러줘야 하죠.
**무엇이 달라지나** — `/remember`로 교훈을 한 번 등록하면(인자 없이 입력해도 방금 작업에서 자동 추론), 이후 `/review`가 모두의 변경분을 그 룰북으로 점수와 함께 검수합니다.

## 문항 2 — 왜 이 문제를 선택했나요? (≤800자)

AI-네이티브 조직의 핵심 문제는 '개발 지식이 개인 세션에 갇히는 것'입니다. 예전엔 코드리뷰·멘토링으로 퍼지던 시니어의 판단이, 각자 AI와 단둘이 일하며 파편화됩니다. 무신사는 이를 공개적으로 보여줍니다.

① 무신사는 AGENTS.md로 '세션 시작 시 규칙을 읽어 모든 생성 코드에 적용'한다고 공개했지만, 이 파일은 수동 관리라 한 사람의 교훈이 자동 반영되지 않습니다.
② ORDER BY의 DATE_FORMAT으로 인덱스가 죽어 p99 350ms→20ms(17배) 개선한 사례, 600줄 god-query, 타입 분기, 단일 Redis 장애 등 비싸게 배운 교훈이 전부 '블로그 글'로만 존재합니다.

즉 무신사 개발자는 '교훈은 있는데 에이전트가 그걸 강제할 형태가 없는' 지점에서 막힙니다. 무신사가 채용 면접에서 Codex 코드를 자체 AI 평가 툴로 채점하듯, 이 플러그인은 그 채점을 현직 개발자의 PR 시점으로 옮깁니다.

## 문항 3 — 플러그인은 어떻게 작동하나요? (≤1000자)

**계층형 룰북** — 규칙을 seed(플러그인 동봉, 무신사 공개 교훈) < org(선택) < repo(.musinsa/rules.yaml, 팀이 키움) 순으로 합쳐 읽고, 아래 계층이 위를 덮어씁니다. 플러그인 코드와 룰 데이터가 분리돼 있어 규칙은 재배포 없이 레포 파일로 자라고, 공유는 git/PR 그대로입니다(별도 서버 불필요).

**두 스킬**
· `/remember` — 교훈을 구조화 규칙(정규식·llm_hint·근거·수정·출처)으로 만들어 룰북에 추가. 규칙을 직접 적으면 그대로, 인자 없이 입력하면 최근 git diff와 대화에서 '무엇을→무엇으로 고쳤는지'를 분석해 규칙을 추론합니다.
· `/review` — 변경분을 룰북으로 검수. 하이브리드 판단: 정규식으로 잡히는 위반은 check.py가 결정론적으로(같은 코드=같은 점수), 설계 냄새는 에이전트가 규칙의 llm_hint로 판단. 100점에서 block −15/warn −8/info −2, block이 하나라도 있으면 게이트 BLOCKED. 출력은 점수 요약 → 근거·수치·출처 보존 결과 → 우선순위 수정 플랜.

**판단 기준** = 룰북 그 자체. 모든 seed 규칙은 무신사 블로그 출처와 수치를 답니다.

**정보가 부족하거나 안 풀릴 때** — ⓐ /remember가 규칙을 명확히 추론 못 하면 멋대로 쓰지 않고 한 가지만 되묻습니다. ⓑ 추론한 규칙은 쓰기 전에 사용자 확인을 받습니다(오추론 방지). ⓒ check.py 의존성이 없거나 실패하면 조용히 넘어가지 않고 그 사실을 밝힌 뒤 수동 판단으로 폴백합니다. ⓓ 애매한 위반은 block이 아니라 info로 표시해 과잉 차단을 피합니다.

## 문항 4 — AI를 어떻게 썼나요? (≤800자)

**AI에 맡긴 일** — 무신사 기술블로그(61개 글 목록화, 핵심 20여 편 정독)·Codex 공개 사례·경쟁 도구를 다중 에이전트로 수집·구조화. 각 규칙의 정규식·수정안 초안과 문서 작성.

**내가 판단한 일** — 무엇을 규칙으로 삼을지(seed 선정), 점수·게이트 설계, '결정론 채점+LLM 판단' 하이브리드 구조, Mode B 추론 도입 여부.

**막힌 지점·해결** — ⓐ 무신사 블로그가 Cloudflare로 막혀 본문을 못 읽음 → Wayback Machine 스냅샷으로 우회해 전문 수집. ⓑ git 배포 설치 시 경로(source/path)가 맞는지 불확실 → 실제 Codex CLI로 marketplace add부터 설치까지 직접 검증.

**받아들이지 않은 AI 제안** — ⓐ '점수를 LLM이 직접 매기게' 하자는 흐름을 거부하고 정규식 결정론 채점을 핵심에 둠(점수가 매번 흔들리면 현업이 신뢰 못 함). ⓑ 리뷰 출력에서 에이전트가 근거를 임의 요약해 '17배' 같은 무신사 수치를 누락 → '요약 금지, check.py 결과 그대로 출력'으로 강제.

## 문항 5 — 어떻게 검증했나요? (≤800자)

**입력→결과 예시** — 입력: `ORDER BY DATE_FORMAT(...)` + `SELECT *` + JOIN 4개인 쿠폰 Mapper와 타입 분기 배송 서비스. 결과: 실제 Codex에서 `Musinsa Readiness 53/100 · BLOCKED`, 위반 5건이 파일:라인·근거(p99 17배)·수정법·무신사 출처와 함께 출력.

**정상/예외 확인** — 정상: 위 BLOCKED, 팀룰(.musinsa/rules.yaml)을 더하면 85/100으로 등록자까지 표시. 예외: 위반 0건 BigDecimal 코드는 100/100 PASS, 스킬명을 안 불러도 '검수해줘' 자연어로 자동 선택됨.

**의심한 것** — LLM 점수가 들쭉날쭉할까 → 결정론 check.py로 고정. URL을 LLM이 흘릴까 → 흘려도 끝 ID로 링크가 열림을 확인.

**고친 점** — 타입분기 정규식이 `getType()` 괄호에서 빗나가 패턴 수정. 수치 누락을 막으려 출력을 3단(요약+verbatim+플랜)으로 재설계.

**아직 부족한 것** — seed 규칙이 6개로 적고, org 공유 계층은 미구현, Mode B 심각도 판단이 실행마다 달라져 확인 단계가 필수.
